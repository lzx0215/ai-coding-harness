# Reviewer Provenance v0.2 Design

## Summary

v0.2 starts with reviewer provenance hardening. The v0.1.1 audit closed immediate metadata gaps by allowing nullable reviewer identity fields and recording the real Claude CLI version and model name when available. v0.2 turns that into a stronger contract for downstream consumers and multi-model review traces.

The current design run still uses `harness_version` and `state_schema_version` `0.1.0` because it was created before the schema migration. The v0.2 implementation slice must update the state schema additively so it accepts both `0.1.0` and `0.2.0`. Existing v0.1.1 run records must remain valid, and new v0.2 implementation runs may then claim `0.2.0` after the schema change is verified.

## Goals

- Define how downstream consumers must handle nullable reviewer metadata.
- Represent reviewer provenance when a review log reports one or more models.
- Add additive `0.2.0` state schema support while preserving existing `0.1.0` run validation.
- Preserve the v0.1 authority boundary: Codex orchestrates, Claude Code reviews read-only, and Harness state remains Codex-owned.
- Keep existing v0.1.1 run records valid.
- Add regression coverage so future changes cannot silently reintroduce unverifiable identity assumptions.

## Non-goals

- Add a dashboard or UI.
- Add async Claude review jobs.
- Let Claude Code mutate files or Harness state.
- Change auth, permissions, secrets, or production configuration.
- Rewrite completed v0.1 or v0.1.1 run records.
- Require every historical review artifact to be migrated immediately.

## Considered Approaches

| Approach | Trade-off | Decision |
| --- | --- | --- |
| Keep only nullable scalar fields | Smallest diff, but multi-model logs remain lossy. | Not enough for v0.2. |
| Replace scalar fields with a new provenance array | Clean model, but breaks existing consumers unless every caller migrates at once. | Too disruptive as the first v0.2 step. |
| Add structured provenance while preserving scalar compatibility fields | Slightly more schema surface, but supports migration and keeps v0.1.1 artifacts readable. | Chosen. |

## Proposed Contract

### State Schema Versioning

The implementation must update `harness/schemas/state.schema.json` so `harness_version` and `state_schema_version` accept `0.1.0` or `0.2.0`.

This is additive compatibility, not a historical rewrite:

- Existing v0.1 and v0.1.1 run records continue to validate unchanged.
- New v0.2 implementation runs may use `0.2.0` only after the schema and tests land.
- Completed historical runs must not be mass-edited just to report `0.2.0`.

Claude review outputs keep the compatibility fields:

- `reviewer`
- `reviewer_model`
- `reviewer_model_version`
- `reviewer_cli_version`

v0.2 adds a structured provenance field:

```json
{
  "reviewer_provenance": {
    "schema_version": "0.2.0",
    "reviewer": "claude-code",
    "cli": {
      "name": "Claude Code",
      "raw_version": "2.1.168 (Claude Code)",
      "version": "2.1.168"
    },
    "models": [
      {
        "name": "glm-5.2[1m]",
        "version": null,
        "source": "modelUsage",
        "usage": {
          "input_tokens": null,
          "output_tokens": null
        }
      }
    ],
    "primary_model": "glm-5.2[1m]",
    "unknowns": [
      "model_version",
      "token_usage"
    ]
  }
}
```

Compatibility scalar fields remain best-effort summaries:

- `reviewer_model` mirrors `primary_model` when available.
- `reviewer_model_version` remains `null` when no verifiable distinct version exists.
- `reviewer_cli_version` keeps the raw CLI version line for audit fidelity.
- `reviewer_provenance.schema_version` and run `state_schema_version` are separate fields, but the implementation must add state schema `0.2.0` compatibility before or in the same commit that allows new output artifacts to emit `reviewer_provenance.schema_version = "0.2.0"`.

### Primary Model Selection

`primary_model` must be deterministic.

Selection algorithm:

1. A model has complete usage only when the adapter can compute a numeric total token count for that model.
2. If at least one model has complete usage, choose the complete-usage model with the largest total token count. Exclude incomplete-usage models from this comparison.
3. If multiple complete-usage models tie, choose the first tied model name in sorted key order.
4. If no model has complete usage but at least one model name is proven, choose the first model name in sorted key order.
5. If no model name can be proven, set `primary_model` to `null` and include `model_name` and `primary_model` in `unknowns`.

The scalar `reviewer_model` mirrors `primary_model`. When provenance exists but `primary_model` is `null`, consumers must treat that as "provenance present, primary model unknown", not as "no provenance".

Invariants:

- If `model_name` appears in `unknowns`, `models` must be empty, `primary_model` must be `null`, `reviewer_model` must be `null`, and `primary_model` must also appear in `unknowns`.
- If `models` is non-empty, `model_name` must not appear in `unknowns`.
- `primary_model` may appear in `unknowns` only when `primary_model` is `null`.

### Unknown Vocabulary

`unknowns` uses a controlled vocabulary:

| Value | Meaning |
| --- | --- |
| `model_name` | No reviewer model name could be proven. |
| `model_version` | A model name was found, but no distinct version could be proven. |
| `cli_version` | Claude CLI version could not be detected. |
| `primary_model` | Multiple or missing model signals prevented a primary model from being selected. |
| `token_usage` | Token counts were absent, incomplete, or not normalized. |

Adapters must not invent new `unknowns` strings without updating this spec and the schema tests.

### Token Usage

This v0.2 slice preserves token usage shape but does not normalize token counts yet.

- `usage.input_tokens` and `usage.output_tokens` may remain `null`.
- Raw model usage entries should be preserved where the adapter can do so safely.
- `token_usage` must be included in `unknowns` when normalized token counts are not available.
- Future token normalization should fill the existing fields rather than require another schema migration.

### CLI Version Parsing

This v0.2 slice preserves the parsed `cli.version` field shown in the example, but no behavior may depend on it yet.

- `cli.raw_version` remains the audit source of truth.
- `cli.version` may be `null` when parsing is not implemented or not reliable.
- Version gating is explicitly deferred until a future feature needs it.
- If CLI version cannot be detected at all, include `cli_version` in `unknowns`.

## Downstream Consumer Rules

Consumers must not assume reviewer identity fields are non-null strings. They should:

- Treat `null` as "not proven", not as failure.
- Prefer `reviewer_provenance.models` when present.
- Fall back to scalar fields for v0.1 and v0.1.1 artifacts.
- Display unknown model versions explicitly instead of inventing one.
- Distinguish "no provenance field" from "provenance exists but primary model is unknown" by checking whether `reviewer_provenance` is present.
- Avoid completion decisions based only on model identity metadata.

## Data Flow

```text
Claude CLI output
-> adapter metadata extraction
-> reviewer_provenance normalization
   -> resolution failure records controlled unknowns instead of throwing (vocabulary in Unknown Vocabulary)
-> compatibility scalar fields
-> output schema validation
-> run state evidence
-> downstream reporting or handoff
```

## Migration Plan

1. Document the v0.2 provenance contract.
2. Update `harness/schemas/state.schema.json` and state schema tests so `harness_version` and `state_schema_version` accept `0.1.0` or `0.2.0`.
3. Validate existing v0.1 and v0.1.1 run records before any adapter can emit new v0.2 provenance artifacts.
4. Add output schema tests for nullable compatibility fields, optional structured provenance, controlled `unknowns`, and a frozen v0.1.1 envelope without `reviewer_provenance`.
5. Add adapter tests for single-model, multi-model, missing-model, incomplete-usage, unknown-vocabulary rejection, and CLI-version parsing cases.
6. Implement provenance extraction without changing Claude permissions.
7. Update README, templates, and handoff guidance for consumer fallback behavior.
8. Add compatibility fixtures proving v0.1.1 envelopes without `reviewer_provenance` still validate.

## Verification Strategy

- Run targeted provenance extraction tests.
- Run full `python -m unittest discover -s tests -v`.
- Validate existing run records with `python -m harness.cli validate`.
- Run an adapter smoke test with fixture metadata.
- If real Claude CLI is available, run one real review smoke and record whether provenance is single-model or multi-model.
- Validate a frozen v0.1.1 review envelope fixture against the new output schema.

## Acceptance Criteria

- v0.1.1 artifacts remain readable and valid.
- New review output schema accepts structured provenance.
- Adapter output records all observed model keys instead of silently choosing only the first key.
- `primary_model` is selected deterministically or recorded as unknown.
- `unknowns` values are validated against the controlled vocabulary.
- Schema validation fails for any `unknowns` value outside the controlled vocabulary.
- Tests cover `null` metadata handling for downstream consumers.
- A frozen v0.1.1 envelope without `reviewer_provenance` validates against the new schema.
- `harness/schemas/state.schema.json` accepts both `0.1.0` and `0.2.0` while existing run records still validate.
- Documentation states exactly what was verified and what remains unverified.
